# SDLC Review Findings — integration/2026-06-26-esv vs main

## Extract shared SSRF host/scheme allowlist guard instead of duplicating it across ESV services

**Severity:** Major (code quality / security-adjacent)

The "scheme must be https AND host must be in an allowlist" guard is implemented separately in three places: `lib/services/bible_lookup_service.dart:120-122`, `lib/services/esv_lookup_service.dart:104-106`, and inlined in `lib/services/esv_audio_cache_service.dart:79,103-108`. This is a security-critical control — duplicating it risks a future fix or host addition landing in only one copy.

**Suggested fix:** extract a shared `assertAllowedHttpsHost(Uri uri, Set<String> allowedHosts)` helper (e.g. `lib/services/net_security.dart`) and have all three services call it.

## Settings screen lets users default to ESV even without an API key configured

**Severity:** Major (correctness / UX)

`lib/screens/settings/settings_screen.dart`'s "Default translation" `SegmentedButton` always offers the ESV segment, regardless of whether `EsvLookupService.isAvailable` is true. `lib/screens/verses/add_verse_screen.dart:38-42` correctly gates ESV behind `isAvailable` and falls back to BSB. A user without an ESV key can set ESV as their default in Settings, then be confused when Add Verse silently substitutes BSB.

**Suggested fix:** gate the ESV segment in Settings behind the same `isAvailable` check (expose it via a provider or static check to avoid instantiating an `http.Client` just to read a getter).

## database_helper.dart imports a service-layer exception type, inverting dependency direction

**Severity:** Minor (architecture)

`lib/database/database_helper.dart:15` imports `LookupException` from `bible_lookup_service.dart` purely to reuse an existing error type when `insertEsvVerse` (lines 278-294) hits the 500-verse cap. This couples the persistence layer to a network-service abstraction.

**Suggested fix:** define a DB-layer-owned exception (e.g. `EsvVerseCapExceededException`) and have callers (`VerseProvider`/`AddVerseScreen`) translate it to a user message.

## Duplicated phase-resume sequences in audio_service.dart

**Severity:** Minor (code quality)

`lib/services/audio_service.dart` re-implements overlapping reference→pause→text playback sequences across `playVerse` (77-105) and the `reference`/`pause` cases of `resume` (127-174). Pre-existing structure, but this branch threaded ESV-audio integration through all three call sites identically, increasing maintenance surface.

**Suggested fix:** refactor to a single `_runFromPhase(_PlayPhase startPhase, Verse verse)` that both `playVerse` and `resume` call into.

## `launchUrl` call to esv.org has no failure handling

**Severity:** Minor

`lib/screens/settings/settings_screen.dart` (~line 305, "ESV.org" `ListTile.onTap`) calls `launchUrl` without `canLaunchUrl` guard or try/catch, inconsistent with the defensive style used elsewhere in this diff (e.g. `EsvAudioCacheService` wraps all network calls).

## UI label "Verse-of-week probability" diverges from underlying field/pref-key naming

**Severity:** Minor (naming / style)

`lib/screens/settings/settings_screen.dart:64,390` renames the user-facing label to "Verse-of-week probability," but `AppSettings.audioInterruptProbability`/`audioInterruptEnabled` (`lib/models/settings.dart:8-9`) and the SharedPreferences keys (`lib/providers/settings_provider.dart:17-18,54-55`) still use "interrupt" terminology. **Severity:** Minor

`lib/models/settings.dart:118-127` and `129-138` repeat an identical "reject far-future timestamps" guard for `lastBackupAt` and `lastVerseAdvanceDate`.

## ESV API key is embedded in the compiled APK (informational, accepted risk)

**Severity:** Informational

The ESV API key is supplied via `--dart-define=ESV_API_KEY=...` at build time, so it's embedded in the compiled APK and extractable via reverse engineering. This is a standard limitation of client-side API keys, not unique to this change, and not fixable without a backend proxy. Given the ESV API's personal/non-commercial distribution terms already on file, this is acceptable for now but should be revisited before any Play Store publish.

## EsvCopyrightFooter announces its collapse/expand state twice via duplicate live regions

**Severity:** High (accessibility — WCAG 4.1.3 Status Messages)

`lib/widgets/esv_copyright_footer.dart:122-128` renders a hidden `Semantics(liveRegion: true, ...)` node alongside the visible content's own `expanded`/`label` semantics (lines 49-52, 72-74), double-announcing every collapse/expand transition, including on first mount when the user did nothing.

**Suggested fix:** drop the separate hidden node; put `liveRegion: true` on the single visible `Semantics` wrapper, and only update the label on a real user-triggered transition (track previous state).

## EsvCopyrightFooter's collapsed toggle is a bare InkWell — no keyboard focus or visible focus ring

**Severity:** High (accessibility — WCAG 2.1.1 Keyboard, 2.4.7 Focus Visible)

`lib/widgets/esv_copyright_footer.dart:48-69` wraps a bare `InkWell` in `Semantics(button: true, excludeSemantics: true)`. `InkWell` has no built-in focus ring and isn't guaranteed keyboard/D-pad focusable the way `IconButton`/`TextButton` are.

**Suggested fix:** replace with an `IconButton`/`TextButton`-based control (matching the expanded state's `IconButton` at line 89), or add `Focus`/`FocusableActionDetector` with an explicit focus-ring decoration.

## EsvCopyrightFooter collapsed tap target may fall below 48×48dp minimum

**Severity:** Medium (accessibility — WCAG 2.5.8 Target Size)

`lib/widgets/esv_copyright_footer.dart:54-67` fixes height at 48px but width is only the intrinsic size of "ESV®" text plus insets, with no `minWidth` guarantee.

**Suggested fix:** wrap in `ConstrainedBox(constraints: BoxConstraints(minWidth: 48, minHeight: 48))` or add symmetric padding.

## Settings "Default translation" ESV subtitle live-region can fire spuriously on screen open

**Severity:** Medium (accessibility — WCAG 4.1.3)

In `lib/screens/settings/settings_screen.dart` (~lines 168-178), the ESV subtitle's `Semantics(liveRegion: true, ...)` is applied unconditionally whenever `defaultTranslation == 'ESV'`, so if that was already the saved setting, opening Settings announces it immediately — competing with the screen's own entrance announcement.

**Suggested fix:** only mark `liveRegion: true` when the value actually transitions during the widget's lifetime, rather than from a conditionally-built static subtitle.

## Default-translation SegmentedButton may compress below target size at narrow widths

**Severity:** Medium (accessibility — WCAG 2.5.8)

In `lib/screens/settings/settings_screen.dart`, the 4-segment `SegmentedButton` (BSB/KJV/WEB/ESV) sits in a `ListTile.trailing` slot, which can compress individual segments below recommended touch-target size at 375px width.

**Suggested fix:** verify rendered width at 375px; if compressed, move the control to its own full-width row below the title, or set `style: SegmentedButton.styleFrom(minimumSize: Size(48, 40))`.

## Redundant Stop/Pause button semantics in review_play_screen.dart

**Severity:** Low (accessibility — WCAG 4.1.2 robustness)

## EsvCopyrightFooter widget reaches into the screen layer by navigating to SettingsScreen directly

## Two different control types used for the same translation-selection choice across screens

**Severity:** Medium (design consistency)

Settings uses `SegmentedButton<String>` for translation choice, while `add_verse_screen.dart` uses an `ActionChip` plus a separate control for the same logical single-select field — two different widget idioms for one concept, and the chip duplicates the "personal use / 500-verse cap" note as part of the tappable control rather than as plain caption text.

**Suggested fix:** standardize on `SegmentedButton<String>` in both places; render the cap/usage note as static caption text beneath it.

## 500-verse ESV cap enforcement in database_helper.dart has zero test coverage

**Severity:** Critical (test coverage)

`lib/database/database_helper.dart`'s `insertEsvVerse` (lines 278-294) is the sole enforcement point for the 500-verse ESV storage cap and uses a transaction specifically to close a TOCTOU race on double-save. No test file (`test/database/database_helper_test.dart`) exists, so neither the cap-exceeded throw, the conflict-ignore insert, nor the race guarantee is verified.

**Suggested fix:** add tests for insert-under-cap, insert-at-cap-throws, and a concurrent `Future.wait` race near the boundary.

## New ESV cap/consent/lookup wiring in add_verse_screen.dart is largely untested (~30-40% coverage)
