Developer Reference

Bible Flashcards has no public API or backend server of its own — it's a fully offline, single-device Flutter app. This page documents the one outbound integration that exists: the optional ESV Bible API client used for verse lookup and audio playback.

Overview

There is no REST API, no server endpoint, and no client-server contract to document for this app — all core data lives in an encrypted local SQLite database (sqflite_sqlcipher) with no network layer. The only network calls this codebase makes for ESV content are made from two service classes against Crossway's ESV API:

FileResponsibility
lib/services/esv_lookup_service.dartFetches verse text from api.esv.org/v3/passage/text/
lib/services/esv_audio_cache_service.dartResolves and caches MP3 recordings from api.esv.org/v3/passage/audio/ (redirects to audio.esv.org)

A parallel, non-ESV lookup path exists for the free bible.helloao.org API (lib/services/bible_lookup_service.dart) — same shape, different host, no API key required. This page focuses on the ESV integration since it has the additional auth, cap, and attribution requirements developers need to get right.

Authentication

Both ESV service classes read the API key via String.fromEnvironment('ESV_API_KEY'), injected at build time:

flutter run --dart-define-from-file=secrets.local

The key is never committed to source control and never written to logs, SharedPreferences, or the database. Each class exposes an isAvailable getter that simply checks the key is non-empty — calling code (Add Verse screen, audio playback) uses this to hide the ESV option entirely rather than rendering a feature that can never succeed:

EsvLookupService({http.Client? client, String apiKey = const String.fromEnvironment('ESV_API_KEY')})
    : _client = client ?? http.Client(),
      _apiKey = apiKey;

bool get isAvailable => _apiKey.isNotEmpty;

The key is sent as an Authorization: Token <key> header — only ever to api.esv.org, never to any redirect target (see Audio Fetch & Cache below).

Text Lookup EsvLookupService

Request

GET https://api.esv.org/v3/passage/text/
    ?q=Romans+8:28
    &include-passage-references=false
    &include-verse-numbers=false
    &include-footnotes=false
    &include-headings=false
    &include-short-copyright=false
Authorization: Token <ESV_API_KEY>
Accept: application/json

Calling it

final service = EsvLookupService();
final result = await service.lookup('Romans 8:28');
// result.reference, result.text, result.translation == 'ESV'

Reference parsing & canonicalization

Before the request fires, the raw input is validated against ^[A-Za-z0-9 :,\-]{1,100}$ and then canonicalized via _canonicalReference(), which splits the input into a book-name span and a chapter:verse(-verse) span using:

static final _refParsePattern =
    RegExp(r'^(.+?)\s+(\d+):(\d+)(?:-(\d+))?\s*$');

The book-name span is resolved to a USFM code via the shared bookNameToUsfm table (lib/utils/book_name_variants.dart) and rewritten to the canonical display name before being sent — so "Rom 8:28" and "Romans 8:28" both resolve to the same query and the same cache key.

SSRF guard

Every request is checked with _assertHttps() before it fires:

void _assertHttps(Uri uri) {
  if (uri.scheme != 'https') throw StateError('Non-HTTPS URL rejected.');
  if (uri.host != 'api.esv.org') throw StateError('Disallowed host: ${'${uri.host}'}');
}

Caching

A 50-entry, instance-scoped, in-memory LRU cache (Map<String, VerseLookupResult>) keyed by "$reference|ESV" avoids re-fetching the same verse during one screen's lifetime. The cache is disposed with the service instance — there is no persistent or cross-session cache for text lookups (the audio cache, below, is different and persists to disk).

Audio Fetch & Cache EsvAudioCacheService

ESV audio uses a deliberate two-request pattern so the API key is never exposed to the CDN host that actually serves the MP3:

  1. Resolve request — GET https://api.esv.org/v3/passage/audio/?q=<reference> with Authorization: Token <key> and followRedirects = false. Expects a 301/302 with a Location header pointing at the CDN.
  2. Fetch request — GET <Location header value>, sent without the Authorization header, only after the target host is checked against an allowlist.
static const _allowedAudioHosts = {'audio.esv.org'};

void _assertAllowedAudioHost(Uri uri) {
  if (uri.scheme != 'https') throw EsvAudioException('Non-HTTPS redirect: ${'${uri.scheme}'}');
  if (!_allowedAudioHosts.contains(uri.host)) {
    throw EsvAudioException('Unexpected audio host: ${'${uri.host}'}');
  }
}

Calling it

final audio = EsvAudioCacheService();
final localPath = await audio.getAudioPath('Romans 8:28');
// localPath points to a cached .mp3 file on disk; play it with audioplayers

Cache key

The cache filename is the SHA-256 hash of the lowercased, trimmed reference — never the raw reference string, which rules out path traversal via a crafted reference value:

String _cacheKey(String reference) {
  final bytes = utf8.encode(reference.toLowerCase().trim());
  return '${'${sha256.convert(bytes)}'}.mp3';
}

Cache location and eviction

Files are written under getApplicationCacheDirectory()/esv_audio/ (app cache dir, excluded from Android Auto Backup). The cache is capped at 250 files; when a new file would exceed the cap, the oldest-modified files are deleted first (_evictIfNeeded). Concurrent calls to getAudioPath() for the same reference share one in-flight Future via an _inFlight map, so simultaneous callers don't trigger duplicate fetches.

Consent & Storage Caps

Consent gate

Both ESV services are gated behind the same SharedPreferences flag, checked before any network call fires:

static const _consentPrefKey = 'esv_lookup_consent_v1';

final prefs = await SharedPreferences.getInstance();
if (prefs.getBool(_consentPrefKey) != true) {
  throw const EsvAudioConsentRequired();
}

This key is isolated from bible_lookup_consent_v1 (used by the non-ESV BibleLookupService) — sharing a single key would silently start sending references to Crossway for users who had only consented to the other provider. Audio playback reuses the same esv_lookup_consent_v1 flag as text lookup rather than prompting separately, on the reasoning that audio only ever plays for verses already saved as ESV — meaning the user already consented to send that exact reference to Crossway once.

500-verse storage cap

Crossway's API terms cap local ESV storage at 500 verses. This is enforced in DatabaseHelper.insertEsvVerse(Verse verse, {int cap = 500}) — not in either service class shown above — via a single transaction that counts existing ESV rows and throws before inserting if the count is already at cap:

await db.transaction((txn) async {
  final count = Sqflite.firstIntValue(await txn.rawQuery(
    "SELECT COUNT(*) FROM verses WHERE translation = 'ESV'",
  ))!;
  if (count >= cap) {
    throw const LookupException('ESV verse limit (500) reached.');
  }
  await txn.insert('verses', verse.toMap());
});

Running the count-check-then-insert inside one transaction prevents a race between two concurrent saves from pushing the stored count past 500.

Error Handling

Both services surface failures through small, typed exception classes rather than generic exceptions, so calling UI code can show specific messages without parsing strings:

ConditionText lookup throwsAudio throws
TimeoutLookupException('Request timed out...')(propagates network error)
404 / not foundLookupException('Verse not found...')n/a
Non-200 / non-redirect statusLookupException('Lookup failed (...)')EsvAudioException('Unexpected status: ...')
Disallowed host on redirectn/aEsvAudioException('Unexpected audio host: ...')
Consent not grantedn/a (consent is checked by the caller before invoking lookup())EsvAudioConsentRequired
Malformed JSON / empty passageLookupException('Could not read response...')n/a

UI-facing behavior differs by feature: a failed text lookup surfaces the exception message directly to the user (it's a user-initiated action they need feedback on). A failed audio fetch is caught by AudioService and silently falls back to on-device text-to-speech — no error state is shown, since audio is a background convenience, not something the user is actively waiting on.
